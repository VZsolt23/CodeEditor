# Architecture

## Layering

Four projects with strictly downward dependencies (Presentation → Application → Core; Infrastructure implements Application interfaces):

```
src/
├── CodeEditor                  Presentation (WPF) — Views, ViewModels, WPF-bound services, DI root
├── CodeEditor.Application     Service interfaces, settings model, framework-free services
├── CodeEditor.Infrastructure  File system, JSON settings persistence, Serilog logging
└── CodeEditor.Core            Language registry, document/workspace primitives
```

- **CodeEditor.Core** — no dependencies. `ILanguageRegistry`/`LanguageRegistry` map file extensions to `LanguageInfo` (id, display name, extensions). Language ids (`csharp`, `typescript`, …) are the stable keys that Roslyn/LSP routing will use in Phases 4–5. `FileSystemEntry` is the workspace primitive the explorer is built on; `SearchQuery`/`SearchMatch`/`SearchSummary` are the Find in Files primitives; `TextSearcher`/`TextSpan` hold the literal-match logic shared by Find in Files and the in-editor find/replace; `DiagnosticItem`/`DiagnosticSeverity` are the Problems-panel primitives Roslyn/LSP will produce.
- **CodeEditor.Application** — interfaces only, plus logic with no framework dependency (`RecentFilesService`, `EditorSettings`, `OutputService` — named output channels with a capped backlog; thread-safe, events may fire on any thread). ViewModels depend on these interfaces, never on Infrastructure types.
- **CodeEditor.Infrastructure** — `JsonSettingsService` (atomic writes: temp file + move), `FileService` (BOM-detecting reads, UTF-8-no-BOM writes), `WorkspaceService` (folder enumeration honoring the `explorerExcludedFolders` setting; `FileSystemWatcher` coalescing bursts into per-directory `DirectoryContentsChanged` events — **raised on background threads**, subscribers marshal to the UI thread; explorer file operations), `SearchService` (workspace-wide literal-text scan on a thread-pool thread, streaming matches through `IProgress<SearchMatch>` — a `Progress<T>` created on the UI thread receives them there; skips excluded folders, binary files via NUL sniff, and files over 16 MB; windows very long lines around the match for display), `RoslynWorkspaceService` (implements `ICodeAnalysisService`: registers `MSBuildLocator` once, discovers `.sln`/`.slnx`/`.csproj` — falling back to a project scan when the solution cannot be parsed (Roslyn 4.14 rejects `.slnx`) — and computes per-document semantic diagnostics against in-editor text, forking and keeping the solution so edits accumulate; a `SemaphoreSlim` serializes workspace access; everything is best-effort and degrades to syntax-only diagnostics), `LspServerConnection`/`LspService` (LSP over stdio via `StreamJsonRpc`: the connection is stream-based and handles the protocol — handshake, full-text sync, publishDiagnostics mapping; the service owns the server process, spawned lazily per workspace from the `typeScriptServerCommand` setting and killed on dispose; all operations best-effort, failures surface via `StatusChanged`), `TerminalService` (hosts the configured shell with redirected stdio; output is pumped in raw chunks — not lines — so prompts without trailing newlines render; cmd.exe sessions are switched to UTF-8 via `chcp 65001`; disposing the service kills all live shell process trees), `LoggingSetup` (composes a Serilog rolling file with `OutputLoggerProvider`, which mirrors Information+ log entries into the Output panel's "Log" channel; factory owned and disposed by `App`). The `OutputService` is constructed in `App.OnStartup` *before* the logger factory (logging needs the channel) and the same instance is registered in DI.
- **CodeEditor (WPF)** — everything that touches WPF, including `ThemeService` (swaps resource dictionaries) and `DialogService` (common dialogs/message boxes), which implement Application-layer interfaces so ViewModels stay UI-framework-free in their dependencies.

## Composition Root

`App.xaml.cs` builds a `ServiceCollection` (all singletons, `ValidateOnBuild` enabled), loads settings, applies the persisted theme, then shows `MainWindow`. Order matters: **settings load before any ViewModel is resolved**, because `EditorOptionsViewModel` copies its initial values from `ISettingsService.Settings` in its constructor. Settings are persisted in `OnExit`.

## ViewModel Structure (CommunityToolkit.Mvvm)

- **`MainViewModel`** — app-level concerns only: window title, theme menu (`ThemeOptionViewModel` items), recent-files menu (`RecentFileViewModel` items), sidebar/bottom-panel visibility, Exit/About/Open-Settings commands, session capture/restore (`CaptureSessionState` fills `EditorSettings.Session`; `RestoreSessionAsync` reopens the workspace, expansion state, and tabs — fire-and-forget from `App.OnStartup` after `Show()`), and the caret-based C# commands (Go to Definition / Find All References / Rename Symbol / Format Document) — multi-location results are displayed by handing `SearchMatch` lists to `SearchViewModel.ShowResults`; rename edits go to open buffers (undoable, dirty) or disk (closed files); format edits apply back-to-front in one `RunUpdate` batch after verifying the buffer still matches the formatted snapshot. Capture happens in `MainWindow.OnClosing` on the *first* pass only: the async close-all flow removes tabs before re-closing, so a later capture would see an empty tab list. Composes the ViewModels below.
- **`DocumentsViewModel`** — the tab manager: open/save/save-as/close/close-all with dirty prompts, auto-save timer. `ActiveDocument` drives command `CanExecute` via `NotifyCanExecuteChangedFor`.
- **`DocumentViewModel`** — one tab: AvalonEdit `TextDocument` (owns the undo stack, so it survives tab switches), file path, dirty flag (set from `TextDocument.Changed`), language, highlighting, caret position. Also the language-service facade for the view: `GetCompletionsAsync`/`GetCompletionChangeAsync` delegate to `ICodeAnalysisService` (null for non-C#), so `EditorView` never touches services directly.
- **`EditorOptionsViewModel`** — singleton shared by all editors: font, word wrap, line numbers, tab width, zoom commands. Writes changes back into the `EditorSettings` model immediately; the file is saved on exit. Zoomed font size is deliberately not persisted.
- **`ExplorerViewModel`** — the sidebar: open/close folder, the root tree node, refresh, and the explorer file operations (prompt → `IWorkspaceService` → reload the affected node → dialog on failure). Subscribes to `DirectoryContentsChanged` and marshals to the UI thread via the dispatcher. After a rename it calls `DocumentsViewModel.HandlePathRenamed` so open tabs track the new path (files and directory subtrees).
- **`FileTreeItemViewModel`** — one tree node. Directory children load lazily on first expansion (a placeholder child keeps the expander visible); `ReloadChildren` re-enumerates while reusing child VMs whose paths still match, preserving expansion state. Per-item commands back the context menu (menus live in separate visual trees, same reasoning as below).
- **`FindReplaceViewModel`** — the inline find/replace panel over the active document. Recomputes matches (via `TextSearcher`) on query/option changes, document edits, and tab switches — but only panel-driven recomputes navigate, so typing in the editor never yanks the caret to a match. Highlight state lives on `DocumentViewModel` (`SearchHighlights`, `CurrentSearchHighlight`) per the per-document-state rule; `SearchHighlightRenderer` (an `IBackgroundRenderer` on the shared editor) paints it, building geometry only for visible spans. Replace All wraps edits in `RunUpdate()` for a single undo step. Navigation uses `NavigateTo(..., focusEditor: false)` so focus stays in the panel.
- **`OutputViewModel`** — the Output panel: channel picker plus the selected channel's text. Channel events arrive on arbitrary threads and in bursts; lines are queued and flushed to the bound `Text` in one dispatcher pass, with a periodic rebuild from the channel snapshot to stay within the backlog cap.
- **`ProblemsViewModel`** — the Problems panel: `Diagnostics` collection plus `SetDiagnostics(source, items)`, which atomically replaces one producer's entries (sorted severity-first). Double-click/Enter navigation reuses `DocumentViewModel.NavigateTo`. Nothing populates it until Phases 4–5.
- **`TerminalViewModel`** — the Terminal panel: a lazily started `ITerminalSession` (first Enter starts the shell in the workspace root), a capped transcript built from output chunks (queued from background threads, flushed in one dispatcher pass, same pattern as Output), Restart/Clear commands, and an exit notice when the shell dies.
- **`SearchViewModel`** — Find in Files: query/options, results grouped into `SearchFileResultViewModel`/`SearchMatchViewModel`, and match navigation. Searches may overlap (a new query cancels and supersedes a running one); a generation counter keeps stale progress reports and status updates out of the UI. Match navigation opens the document and calls `DocumentViewModel.NavigateTo`, which sets the observable `PendingNavigation` property — the shared `EditorView` consumes and clears it once its `Document` binding matches the requesting ViewModel (triggered by `DataContextChanged`, `Editor.DocumentChanged`, or the property change itself), so requests made before the tab finished switching are not lost.

Menu items that need per-item commands (themes, recent files) get small item-ViewModels with their own command instead of `RelativeSource` binding tricks — menu popups live in separate visual trees where ancestor bindings fail.

**`CSharpDiagnosticsCoordinator`** (Services, not a ViewModel) is the glue between the editor and C# analysis: it loads/unloads the Roslyn workspace on `WorkspaceChanged`, watches open C# documents (attach/detach on the tab collection), debounces text changes (700 ms `DispatcherTimer`), and pushes per-document diagnostics into `ProblemsViewModel` under the "csharp" source. It is resolved explicitly in `App.OnStartup` **before** session restore, or it would miss the `WorkspaceChanged` that restore raises. Load status flows to the status bar via `ICodeAnalysisService.StatusChanged` → `MainViewModel.StatusText`.

**`LspDiagnosticsCoordinator`** is its LSP twin for TypeScript/JavaScript documents: it forwards didOpen/didChange (500 ms debounce)/didClose to `ILspService` and — since LSP *pushes* diagnostics rather than answering pulls — routes `DiagnosticsPublished` events (marshalled to the UI thread) into `DocumentViewModel.Diagnostics` and the Problems panel under the "typescript" source. Also resolved in `App.OnStartup` before session restore.

## Views

- **`MainWindow.xaml`** — menu, toolbar (a styled `StackPanel` with `FocusManager.IsFocusScope="True"` so routed commands like Undo reach the focused editor), status bar, workbench grid (sidebar tabs EXPLORER/SEARCH | splitter | editor+bottom panel). Code-behind holds only view logic: panel collapse/restore (grid metrics aren't binding-friendly), the async close-confirmation flow (`OnClosing` cancels, awaits the save prompts, then re-closes with `_closeApproved`), explorer/search-results input wiring (double-click/click opens, Enter opens/toggles, folder expansion stays native), and focusing the search box on Ctrl+Shift+F (`FindInFilesRequested`).
- **`InputDialog.xaml`** — themed single-line prompt used by `DialogService.ShowInputDialog` (explorer new file/folder/rename; preselects the name stem so renames overtype it).
- **`EditorView.xaml(.cs)`** — hosts AvalonEdit. **One instance is reused across tab switches** (it sits in the `TabControl.ContentTemplate`), so all wiring reacts to `DataContextChanged`. Code-behind wires what XAML can't bind: caret→ViewModel, indentation strategy by language, tab width, selection/current-line brushes via `SetResourceReference`, Ctrl+wheel zoom, the find-match and squiggle background renderers (fed from `DocumentViewModel.SearchHighlights`/`Diagnostics`), the hover tooltip (`TextView.MouseHover`; Quick Info fetched async and merged with diagnostics, a token discards stale results), signature help (`(`/`,`/Ctrl+Shift+Space → `OverloadInsightWindow` fed by `RoslynOverloadProvider`), semantic colors (`SemanticHighlightColorizer` — a line transformer added after AvalonEdit's highlighting colorizer so classified spans override the `.xshd` colors; fed from `DocumentViewModel.SemanticHighlights`, mapped to `Brush.Syntax.*` theme keys), and code completion (`TextEntered`/Ctrl+Space → `CompletionWindow`; `RoslynCompletionData.Complete` blocks briefly on `GetCompletionChangeAsync` — safe because the service runs on the thread pool with `ConfigureAwait(false)` throughout — and replaces the live completion segment, so characters typed while the window was open are handled).

## Theme System

`App.xaml` merges two dictionaries: the active theme (index 0) and `Themes/Styles.xaml`. `ThemeService.ApplyTheme` replaces the theme dictionary at runtime; every color in `Styles.xaml` and the views is a `DynamicResource`, so the switch is live.

**Contract:** every theme dictionary must define the exact brush-key set in `DarkTheme.xaml` (`Brush.Window.*`, `Brush.Menu.*`, `Brush.Tab.*`, `Brush.Editor.*`, …). New themes = new dictionary + registration in `ThemeService`; the View menu picks them up automatically.

`Styles.xaml` re-templates `MenuItem` (two templates: top-level and popup items, with check glyphs and gesture text), `TabControl`/`TabItem`, toolbar/tab-close buttons, `Separator`, `GridSplitter`, `ToolTip` — because WPF's default chrome ignores dark backgrounds.

## Conventions and Pitfalls

- **`.slnx` solution format** — build with `dotnet build CodeEditor.slnx`; there is no `.sln`.
- **Namespace shadowing:** the `CodeEditor.Application` namespace shadows `System.Windows.Application` inside any `CodeEditor.*` namespace. Always write `System.Windows.Application` fully qualified in the WPF project.
- **Sync-over-async at startup/exit:** `App` blocks on `LoadAsync`/`SaveAsync` via `Task.Run(...).GetAwaiter().GetResult()`. Anything awaited on such paths must use `ConfigureAwait(false)` **including implicit `await using` disposals** (`await using (stream.ConfigureAwait(false))`). A missed one deadlocked `OnExit` once — see `JsonSettingsService`.
- **Highlighting fallbacks:** AvalonEdit has no TypeScript/JSX definitions; `EditorHighlighting` maps those to JavaScript, MSBuild files to XML, Razor to HTML. This is a stopgap until semantic highlighting (Phases 4–5).
- `Directory.Build.props` sets `Nullable`, `ImplicitUsings`, `LangVersion latest` for all projects.
- Smoke test (used to verify the shutdown path): launch the exe, `taskkill /PID <pid>` **without** `/F` (sends `WM_CLOSE`); a healthy run logs `Code Editor exited` and rewrites `settings.json`.
